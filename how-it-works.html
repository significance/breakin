[ SECTION 1 ] THE BIG PICTURE

How does a 1993 DOS game run inside your web browser?

BREAKIN.EXE was compiled for Intel 286 real-mode DOS. It expects a VGA card, a mouse driver, and optionally a Sound Blaster 16. None of these exist in your browser.

The trick: we run a complete DOS emulator compiled to WebAssembly, executing every single x86 instruction of the original game in a virtual machine inside your browser tab.

[ SECTION 2 ] THE .JSDOS BUNDLE

What is a .jsdos file?

It's a ZIP archive with a special structure. Inside you'll find the original DOS game files plus a .jsdos/dosbox.conf configuration file that tells the emulator how to set up the virtual machine.

Our bundle breakanoid.jsdos (172KB) contains 34 files: 

# You can unzip it like any ZIP file:
$ unzip breakanoid.jsdos -d breakin-files/

# The magic is in the config:
breakin-files/
  .jsdos/
    dosbox.conf    <-- emulator configuration
  BREAKIN.EXE       <-- the game! (93KB)
  BREAKIN.ANI       <-- animation data (32KB)
  EXTRA.ANI         <-- extra weapon animations
  ENEMY.SPR         <-- enemy sprite sheet
  BLOCKS.BUP        <-- brick graphics (32KB)
  *.PIC             <-- 256-color title screens (65KB each)
  *.FNT             <-- bitmap font data
  *.VOC             <-- Sound Blaster voice samples
  TRACK2.MUS        <-- FM synthesis music
  TESTLVL.*         <-- level data + hit maps
  TOP100.ASG        <-- high score table
NOTE: The .PIC files are raw 320x200 VGA framebuffer dumps -- 64,768 bytes each (320 * 200 = 64,000 pixels + 768 bytes of palette data). The palette is appended as 256 RGB triplets. This was standard practice in the DOS shareware era -- no PNG, no JPEG, just raw video memory!

The JavaScript code that loads this bundle: 

Dos(document.getElementById("dos"), {
  url: "breakanoid.jsdos",        // the ZIP bundle
  autoStart: true,                // skip the "click to start" screen
  pathPrefix: "assets/emulators/", // where to find WASM files
  onEvent: (event, ci) => {
    if (event === "ci-ready") {
      // emulator is running! BREAKIN.EXE has started.
      // ci = command interface for mute/unmute etc.
      dosCI = ci;
      hideLoader();
    }
  }
});

That's it. One function call. js-dos handles everything else: fetching the ZIP, extracting files into a virtual filesystem, parsing dosbox.conf, booting the emulated DOS, and running the EXE.

[ SECTION 3 ] DOSBOX CONFIGURATION

The .jsdos/dosbox.conf file is the heart of the emulation setup.

It configures the virtual hardware to match what BREAKIN.EXE expects to find on a 1993 PC: 

# .jsdos/dosbox.conf

[sdl]
fullscreen=false
autolock=true          # auto-capture mouse on click

[dosbox]
machine=svga_s3       # emulate S3 Trio SVGA card
                       # (BREAKIN only needs VGA 320x200x256)

[cpu]
core=auto             # dynamic recompiler when available
cputype=auto
cycles=3000           # ~3000 instructions per millisecond
                       # roughly equivalent to a 286 @ 12MHz

[sblaster]
sbtype=sb16           # Sound Blaster 16 emulation
sbbase=220            # I/O base address (standard)
irq=7                 # interrupt request line
dma=1                 # 8-bit DMA channel
hdma=5                # 16-bit DMA channel

[mixer]
rate=44100            # output sample rate in Hz

[autoexec]
@echo off
mount c .              # mount bundle root as C:\
c:
BREAKIN.EXE            # launch the game!

How the cycle count works: DOSBox doesn't run at a real clock speed. Instead, it executes cycles instructions per millisecond. 3000 cycles/ms gives roughly 286-12MHz performance -- exactly what the original README specifies as the minimum requirement.

HARDWARE EMULATED:
CPU: Intel 286 equivalent (3000 cycles)
GPU: S3 Trio SVGA (VGA Mode 13h: 320x200, 256 colors)
Sound: Creative Sound Blaster 16 (IRQ 7, DMA 1, base 220h)
Mouse: PS/2 compatible (via browser mouse events)
Disk: Virtual C:\ drive (from ZIP contents)

The [autoexec] section works just like AUTOEXEC.BAT -- it mounts the game files as drive C: and immediately runs BREAKIN.EXE. The user sees the game title screen within seconds of the page loading.

[ SECTION 4 ] WEBASSEMBLY EMULATION ENGINE

The emulator is DOSBox, compiled from C++ to WebAssembly.

The assets/emulators/ directory contains the entire emulation runtime:

FILE SIZE PURPOSE
wdosbox.wasm 1.4 MB Standard DOSBox -- the actual emulator binary
wdosbox.js 101 KB Emscripten JS glue code for wdosbox.wasm
wdosbox-x.wasm 6.4 MB Extended DOSBox-X (more hardware, not used here)
wdosbox-x.js 216 KB Glue code for DOSBox-X variant
wlibzip.wasm 110 KB libzip compiled to WASM -- unpacks .jsdos bundles
wlibzip.js 73 KB Glue code for libzip
emulators.js 70 KB Abstraction layer: loads correct WASM modules

The compilation pipeline that created these files: 

# DOSBox is written in C++ (~200K lines)
# Emscripten compiles it to WebAssembly:

DOSBox C++ source
       |
       v
  [Emscripten compiler (emcc)]
       |
       +--> wdosbox.wasm   (compiled binary)
       +--> wdosbox.js     (JS runtime/glue)

# The .wasm file contains compiled machine code that runs
# at near-native speed in the browser's WASM VM.
#
# It handles:
#   - x86 CPU instruction decode + execute
#   - VGA framebuffer rendering
#   - Sound Blaster DSP + FM synthesis (OPL3)
#   - DMA transfers, IRQ handling, PIC, PIT timers
#   - FAT filesystem emulation
#   - PS/2 mouse + keyboard controller
WHY WASM? JavaScript is too slow for cycle-accurate CPU emulation. WebAssembly runs at near-native speed -- close to a C++ binary compiled for your actual CPU. The 1.4MB wdosbox.wasm replaces what would have been a 50MB+ native DOSBox install. The game thinks it's running on a real 286 with a Sound Blaster card. It has no idea it's inside a browser.

The boot sequence: 

// 1. js-dos.js fetches breakanoid.jsdos (ZIP)
// 2. wlibzip.wasm extracts the ZIP in-memory
// 3. dosbox.conf is parsed for hardware config
// 4. wdosbox.wasm is loaded + initialized
// 5. Game files are mounted as a virtual C:\ drive
// 6. [autoexec] runs: mount c . && BREAKIN.EXE
// 7. VGA output is rendered to a <canvas> element
// 8. Audio samples are pushed to Web Audio API
// 9. Mouse/keyboard events are translated to DOS interrupts

[ SECTION 6 ] INPUT & AUDIO PIPELINE

How does your mouse control a DOS game?

When you move your mouse over the game canvas, this chain fires: 

Browser mousemove event
    |
    v
js-dos event handler
    |
    v
Emscripten bridge (JS --> WASM function call)
    |
    v
DOSBox PS/2 mouse controller (in WASM)
    |
    v
DOS mouse driver interrupt (INT 33h)
    |
    v
BREAKIN.EXE reads mouse position --> moves paddle

The game's original mouse code is unchanged. It calls INT 33h (the DOS mouse interrupt) exactly as it did in 1993. DOSBox's emulated mouse controller translates browser events into the same hardware signals a real PS/2 mouse would generate.


Audio: Sound Blaster emulation

BREAKIN uses two audio systems:

SOUND TYPE HOW IT WORKS
.VOC samples
(BONG.VOC, SMASH.VOC...)		 Digitized sound effects played through Sound Blaster DSP via DMA. The game writes PCM samples to the SB's DMA channel. DOSBox intercepts these writes, mixes the audio, and feeds it to the Web Audio API.
FM Music
(TRACK2.MUS)		 Music played through the OPL3 FM synthesis chip on the Sound Blaster. DOSBox emulates the Yamaha YMF262 chip register-by-register, generating audio samples in WASM, output via Web Audio API at 44.1kHz.

The page provides a mute toggle:

let dosCI = null;  // command interface from js-dos
let muted = false;

function toggleSound() {
  if (dosCI) {
    if (muted) {
      dosCI.unmute();
      muted = false;
    } else {
      dosCI.mute();
      muted = true;
    }
  }
}

Keyboard shortcuts are handled by a single listener:

document.addEventListener("keydown", (e) => {
  // ESC closes the docs overlay
  if (e.key === "Escape" && docOverlayOpen) {
    toggleDocs();
    return;
  }
  // F = fullscreen, H = help/docs
  if (e.key === "f") goFullscreen();
  if (e.key === "h") toggleDocs();
});
INPUT CAPTURE: When you click on the game canvas, the browser locks the mouse pointer via the Pointer Lock API. This lets DOSBox receive raw mouse deltas (just like a real mouse driver) rather than absolute screen coordinates. Press ESC to release the capture.
